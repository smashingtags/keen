# Contributing to Keen

Thanks for your interest in contributing to Keen! This document covers the process for contributing to the project.

## Before You Start

Before submitting code, open an issue describing the problem and your proposed approach. PRs without a linked issue will be closed.

For new features, open an issue first so we can discuss the design before you invest time building it.

Bug fixes are prioritized over new features.

## A Note on Code Generation

This codebase is partially generated by CLI Printing Press. If you find a systemic issue (e.g., a pattern repeated across many files), it may be better fixed upstream in the generator rather than patched in individual files. Open an issue and we'll figure out the right place to fix it.

## Development Workflow

1. Fork the repository
2. Create a feature branch from `main`:
   ```bash
   git checkout -b feat/your-feature
   ```
3. Make your changes
4. Commit using conventional commit format (see below)
5. Push to your fork and open a pull request

## Building

```bash
make build       # build for your current platform
make build-all   # cross-compile for all supported platforms
```

## Testing

```bash
make test
```

## Linting

```bash
make lint        # runs golangci-lint
```

Make sure lint and tests pass before submitting a PR.

## Commit Message Format

This project uses [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>: <description>

[optional body]
```

Types:

- `feat` -- new feature
- `fix` -- bug fix
- `docs` -- documentation only
- `chore` -- maintenance, deps, CI, tooling

Examples:

```
feat: add bulk issue export to CSV
fix: handle pagination for large Confluence spaces
docs: update MCP server configuration examples
chore: bump golangci-lint to v2.1
```

## PR Review Schedule

I review PRs on Tuesdays and Fridays. If your PR is urgent (security fix, blocking bug), mention it in the issue and I'll try to get to it sooner.
